# CONTENT_STYLE_STANDARD(内容风格标准)

2026-07-11 制定。**以后所有文章与工具介绍的撰写、改写都按本文件执行。**

与其它标准的分工:`ARTICLE_CONTENT_STANDARD.md` 管文章的**结构**(速览框/来源披露/编号小节/WHY/KEY);本文件管**句子和标题怎么写**——用什么口吻、什么算 AI 味、什么算真实体验感。两份都要过。

## 目标口吻:用过的人向朋友介绍

想象的作者是:一个真的把工具用了一段时间的普通用户,朋友问起来,他讲"这东西哪里好用、哪里会坑你、什么情况别用"。**不是导师,不是客服,不是百科,也不是一个反复强调自己的"我"。**

## 一、AI 味黑名单(出现即改)

以下是本站现有内容里实际出现过的 AI 味模式,写作和审稿时逐条对照:

### 1. 叙述者排比标题(最严重)

小标题**禁止出现"我"**,禁止用叙述顺序当标题。标题必须携带信息或结论,读者扫一眼标题就能带走东西。

- 反例(来自现有文章):「我先看隐私 / 我再看商用 / 我最后看免费版」「我先看来源 / 我再看授权 / 我还会看版本」
- 正例:「上传之前,先想清楚这份资料能不能见人」「"AI 生成"不等于"能拿去卖"」「免费版是试吃,不是正餐」

### 2. "我"当骨架

第一人称可以出现,但只能当佐料——讲一个具体操作、具体决定时自然带出。全文以"我会…我就…我更愿意…"推进,是 AI 模仿人味的典型写法,读者无法代入。

- 量化红线:正文段落里"我"字连续 3 段开头出现 = 必改;通篇"我"密度明显高于"你"= 结构反了。
- 多数句子的主语应该是**工具、任务或"你"**:"你上传的是合同,就先停一下" 比 "我会先停一下" 更能带入。

### 3. 模板句式复用

同一批内容里反复出现的句型,第二次出现就是模板:

- 「X 最大的价值不是 A,而是 B」(现有 20 个工具里出现 5+ 次)
- 「X 不等于 Y,真正重要的是 Z」
- 「它也不是没有坑」「但它也不是万能的」
- summary 统一句式「X 适合…,但不适合…」(现有工具几乎全部命中)
- 对称三连/四连排比(「写标题、改句子、整理公开资料」)一篇最多一处,多了就是机器节奏。

### 4. 全站同构

每篇文章都是「先给结论 → 三个维度 → 我自己的判断法」,每个工具都是「定位一段 → 对小白来说一段 → 但是一段」——结构可以相似(结构标准管),但**开头方式、切入角度、收尾方式必须一篇一个样**。连续两篇用同一个开头套路即违规。

### 5. 空泛概括代替具体细节

「任何一项都可能卡住你」「很多人容易踩坑」这类句子没有信息量。每个论点必须落到至少一个**具体细节**上(见下)。

## 二、真实体验感从哪里来(不许编故事)

**红线:本站不虚构。**"真实体验风格"绝不等于编造"我上周差点把合同传上去"这类没发生过的亲历。体验感的合法来源按优先级:

1. **可核实的产品事实细节**:免费版导出分辨率是多少、免费额度几次、哪个按钮藏在哪里、第一次打开会看到什么、哪类图片处理效果差(毛发/透明物体)。这些查官网/实测就能拿到,写进去立刻有"用过的人"的味道。
2. **具体到任务的场景**(用"你"或无主语):「做小红书封面,套模板十分钟能出图;要改别人给的 PSD,它就吃力了」——场景具体到任务、耗时、结果,但不伪装成某次亲历。
3. **真实操作路径**:「从官网下载 → 首次启动要在系统设置里手动放行」这类只有走过一遍流程才写得出的步骤。
4. 确实做过的实测(如本站导入工具时的核实过程)可以写,注明性质。

写不出以上任何一种具体细节的论点,删掉或降级成一句话,不要用排比撑篇幅。

## 三、工具介绍的专项规则

- **summary(一句话简介)**:禁用统一模板。写这个工具最独特的一件事 + 最需要知道的一个限制,句式自由。
- **description**:至少包含一条"实际用的时候才知道"的细节(见上,第 1/3 类)。两种现存的坏声音都不要:①"对小白来说…"导师腔(现有前 10 个工具);②README 翻译腔"是一款开源的…采用 XX 协议…请以官方页面为准"(现有后 10 个工具)——协议、平台这些事实保留,但要有至少一段像人话的使用感受。
- **pros/cons**:不必凑 5×5 对称;3 条具体的好过 5 条空泛的。每条尽量带名词细节(格式名、场景名、限制数值),禁止"功能强大""比较友好"这类空评价。
- **risk_notice**:讲清楚一件真正会发生的坏事,不要道德说教式全覆盖。

## 四、文章的专项规则(叠加在结构标准之上)

- 标题(h2)按本文件第一条写:带结论、带画面,不带"我"。
- [速览] 要点写具体结论,不写"本文介绍了…"。
- [KEY] 高亮框里放的必须是可执行判断(「上传前先问:这份资料能不能见人」),不是口号。
- 每篇至少一处具体细节锚点(数字、名称、路径、限制),否则不发布。
- 提到站内已收录的工具时,自然写出工具名(将来可加站内链接),不要用"轻量工具""某些平台"含混带过。

## 五、发布前检查清单

- [ ] 所有小标题无"我",且每个标题单独携带信息
- [ ] 正文"你/工具/任务"做主语的句子多于"我"做主语的句子
- [ ] 黑名单句式零命中("最大的价值不是…而是…"等)
- [ ] 至少 1 个可核实的具体细节(数值/路径/限制/场景),文章类至少 2 个
- [ ] 排比不超过一处
- [ ] 开头和收尾方式与最近发布的一篇不同构
- [ ] 没有编造的亲历;所有"体验细节"可以指出核实来源
