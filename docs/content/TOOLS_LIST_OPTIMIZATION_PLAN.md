# 工具列表页信息密度与筛选体验优化规划

## 1. 当前背景

工具库已经从 5 个工具扩展到 10 个工具。

当前工具覆盖：

- AI 问答：ChatGPT
- 设计制图：Canva
- 在线修图：Photopea
- 笔记整理：Notion
- AI 编程：Cursor
- UI / 产品设计：Figma
- 短视频剪辑：CapCut
- 图片抠图：remove.bg
- 图片压缩：TinyPNG
- 知识管理：Obsidian

当工具数量较少时，用户可以直接浏览完整列表。
但工具数量达到 10 个以后，列表页需要承担更强的“发现”和“筛选”能力。

工具列表页的目标不是把所有信息堆出来，而是让小白用户快速判断：

- 我现在要解决什么问题？
- 哪个工具适合我？
- 哪个工具我应该先点进去看？
- 这个工具是不是免费的？
- 有没有官网吗？
- 有没有风险提醒？

## 2. 当前页面可能存在的问题

根据当前阶段内容增长，工具列表页可能面临这些问题：

第一，首屏可能太偏展示，工具卡片出现得不够早。  
用户进入 `/tools` 后，应该尽快看到工具，而不是先浏览过长的 Hero 和说明模块。

第二，筛选区可能偏重。  
筛选是辅助，不应该比工具列表本身更抢眼。移动端尤其要避免筛选区域占太多高度。

第三，工具卡片信息需要更像“选择建议”。  
小白用户不只是想看工具名称，而是想知道这个工具适合做什么、适合谁、不适合谁。

第四，分类筛选需要更贴近用户任务。  
单纯按“AI 工具 / 在线工具 / 效率工具”分类还不够直观。用户更容易理解的是“写作问答 / 做图设计 / 图片处理 / 视频剪辑 / 笔记整理 / 编程辅助”。

第五，移动端连续浏览体验需要优化。  
手机端应该让用户连续刷工具卡片，不要每张卡片都太高，也不要每个区域都像大卡片堆叠。

## 3. 优化目标

本次工具列表页优化要达到：

- 首屏更快露出工具内容。
- 搜索和筛选更轻。
- 工具卡片更容易扫读。
- 移动端工具卡片更紧凑。
- 分类更像用户任务，而不是后台字段。
- 保持高级、干净、简洁的视觉风格。
- 不增加复杂功能。
- 不引入用户系统和互动功能。

## 4. 页面结构建议

建议 `/tools` 页面结构调整为：

```txt
顶部导航
工具库 Hero
快速任务入口
搜索与筛选
工具列表
底部说明 / 广告 / Footer
```

但每一块都要克制，不能做得太重。

### 推荐结构

```txt
1. Hero：一句话定位 + 简短说明
2. 快速任务入口：按用户任务展示 5-6 个入口
3. 搜索与筛选：搜索框 + 分类 chips
4. 工具列表：10 个工具卡片
5. 底部：轻量说明与 Footer
```

## 5. Hero 优化方向

Hero 不要太高。

当前工具数量已经足够，用户进入页面后应该尽快看到工具列表。

Hero 建议：

- 桌面端高度控制在中等，不要大面积空白。
- 移动端 Hero 上下 padding 控制在 40px 左右。
- 标题直接表达站点价值。
- 副标题减少空话。

推荐文案方向：

```txt
找到真正适合你的工具
不只收集链接，也告诉你适合谁、哪里坑、怎么选。
```

或者：

```txt
工具不是越多越好，适合你才重要
这里整理 AI、设计、图片、视频、效率和编程工具，重点说清楚小白该怎么选。
```

不要使用：

```txt
探索数字时代的效率革命
赋能未来工作流
开启你的智能生产力之旅
```

这类空泛文案。

## 6. 快速任务入口

工具列表页应该增加“按任务找工具”的感觉。

推荐任务入口：

- 写作问答
- 做图设计
- 图片处理
- 视频剪辑
- 编程辅助
- 笔记整理

每个任务入口可以是轻量 chip 或小卡片，不要做复杂模块。

示例：

```txt
我想写作 / 问答
我想做封面 / 海报
我想修图 / 抠图 / 压缩
我想剪短视频
我想写代码
我想整理笔记
```

点击后可以复用现有分类筛选或搜索逻辑。
如果现有代码不支持任务筛选，第一轮可以只做视觉入口，不新增复杂状态逻辑。

## 7. 搜索与筛选区

搜索和筛选区要轻量化。

建议：

- 搜索框置于工具列表上方。
- 分类 chips 横向排列。
- 移动端允许横向滑动 chips。
- 筛选区不要使用过高的大卡片。
- 搜索框 placeholder 要更像用户语言。

推荐 placeholder：

```txt
搜索工具名称、用途，比如：抠图、剪视频、写代码
```

分类筛选建议显示：

```txt
全部
AI 工具
设计工具
图片处理
视频剪辑
效率笔记
编程工具
```

如果真实分类名称不完全一致，以数据库分类为准。
不要为了筛选强行改数据库 schema。

## 8. 工具卡片优化方向

工具卡片要更像“小白判断卡”，而不是数据库字段展示。

每张卡片建议展示：

```txt
工具名称
一句话 summary
分类
免费 / 付费状态
开源状态
适合场景短句
官网按钮 / 查看详情按钮
```

卡片不建议展示太多字段：

- 不要把优点、缺点、风险全部塞到列表卡片。
- 不要展示长段 description。
- 不要展示过多标签。
- 不要出现网盘下载按钮。
- 不要把卡片做得像详情页缩略版。

卡片重点应该是：

```txt
这个工具大概是干什么的？
我是不是该点进去？
```

## 9. 移动端工具卡片设计

移动端卡片要紧凑。

建议：

- 卡片 padding：8px - 20px。
- 卡片圆角：12px - 26px。
- 标题字号：16px - 22px。
- summary 字号：14px - 15px。
- 行高：1.6 - 1.7。
- 卡片之间间距：4px - 18px。
- 标签不要超过 3 个。
- 按钮不要过多。

移动端卡片结构建议：

```txt
分类 / 免费状态
工具名称
一句话简介
适合：短句
查看详情
```

不建议在移动端卡片里出现：

- 大量标签
- 长 description
- 多个按钮
- 网盘下载
- 复杂 hover 效果

## 10. 桌面端工具卡片设计

桌面端可以保持网格：

- 2 列或 3 列，根据当前宽度决定。
- 卡片高度尽量一致。
- 分类和状态放在顶部。
- 工具名称和 summary 放中间。
- 底部放“查看详情 / 官网”。

桌面端不需要过度压缩，但也不能太松。

## 11. 排序建议

第一轮不做复杂排序算法。

可以采用固定排序原则：

1. 常用基础工具靠前：

   - ChatGPT
   - Canva
   - Figma
   - Photopea
   - CapCut
2. 专项工具随后：

   - remove.bg
   - TinyPNG
3. 效率和开发工具：

   - Notion
   - Obsidian
   - Cursor

也可以沿用当前数据库排序。
第一轮不新增排序字段，不改 schema。

## 12. 空状态设计

当搜索或筛选没有结果时，不要只显示“暂无内容”。

建议文案：

```txt
暂时没找到匹配工具
可以换个关键词试试，比如：图片、视频、AI、笔记、编程。
```

并提供“清空筛选”按钮。

## 13. 不做的事情

本阶段明确不做：

- 用户收藏
- 点赞
- 评论
- 转发
- 排行榜
- 用户登录
- 工具评分
- 后台批量导入
- AI 自动生成内容
- 自动抓取官网
- 网盘下载入口
- 修改数据库 schema

## 14. 第一轮代码改造优先级

后续代码改造建议按这个顺序：

### 第一优先级：移动端工具列表页密度

- 压缩 Hero 高度。
- 让工具卡片更早出现。
- 收紧筛选区高度。
- 移动端卡片更紧凑。

### 第二优先级：工具卡片信息重组

- 强化 summary。
- 展示分类和免费状态。
- 展示“适合场景短句”。
- 弱化多余标签和装饰。

### 第三优先级：搜索与分类筛选

- 搜索框更清晰。
- 分类 chips 横向滑动。
- 空状态更友好。

### 第四优先级：桌面端网格微调

- 保持高级感。
- 卡片高度更统一。
- 减少无意义留白。

## 15. 验收标准

改造完成后，`/tools` 必须满足：

- 线上和本地都显示 10 个工具。
- 移动端第一屏能更快看到工具内容。
- 搜索和筛选区不压迫页面。
- 工具卡片更容易扫读。
- 卡片不是详情页缩略版。
- 不显示 Open Design / Raycast。
- 不显示未分类。
- 不显示 undefined。
- 不显示 null。
- 不出现字符拆分。
- 不出现 duplicate key。
- 不出现网盘下载入口。
- 旧工具和新工具详情页不受影响。
