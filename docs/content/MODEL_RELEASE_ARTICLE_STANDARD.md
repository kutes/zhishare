# 模型/产品发布类文章规范 v1

2026-07-17 制定。适用于「某厂商发布了新模型/新产品」这类时效性解读文章（首篇样本：Kimi K3）。
本规范是 `docs/ARTICLE_CONTENT_STANDARD.md` 的补充——通用结构（速览框、编号小节、lead 导语、图注带来源、收束句）全部沿用，此处只定义发布类文章特有的规则。

## 1. 三层信源模型（每篇必须分层）

| 层 | 内容 | 要求 |
| --- | --- | --- |
| 官方口径 | 厂商博客/发布页的功能、架构、跑分、价格 | 必须直接抓取官方原文核实，逐个数字对照；正文里明确标注「厂商自评」 |
| 独立检验 | 第三方评测机构与媒体的数据 | 必须去原始出处（如 The Decoder、Artificial Analysis）核实后才能引用；**禁止从参考文章转抄二手数据**——自己没打开过的来源不进文章、不进脚注 |
| 交叉核对 | 涉及其他厂商的价格/参数对比 | 用对方官方资料核对（如 Anthropic 定价查官方文档）；核不到的数字宁可不放 |

## 2. 口径披露（三处，缺一不可）

1. **开篇 ⚑ 来源披露**（沿用通用标准）：一句话说清「本文基于 X 整理，跑分为厂商自评口径；独立检验一节数据来自 Y」。
2. **正文内随处标注**：厂商自评的跑分表、内部评测，就地写明「自评口径」「参考着看」；对比表要交代测试条件（如统一用最高推理档、对手环境不同等）。
3. **结尾来源脚注**（发布类特有，放全文最后一节）：固定格式——

   > 来源：{厂商} 官方博客《{标题}》（{日期}）；独立评测与对比数据来自 {实际查证过的第三方}；{交叉核对说明}。除{独立检验小节名}外，性能数据为厂商自评口径；{时效性声明，如：技术报告与模型权重截至发稿尚未放出}。

   只列**实际打开核实过**的来源。参考文章（如别家解读站）不是信源，不列入。

## 3. 结构模板（5-7 节，按需取舍）

1. **发布**：什么、何时、能用了吗（lead 先给结论）+ [WHY] 为什么值得看 + [STATS] 关键数字
2. **架构/原理**（可选）：用小白比喻讲清技术点，一节最多讲 3 个概念
3. **跑分**：精选对比表（[KV] 或列表），必配「哪里第一/哪里还差」的双栏总结，测试条件先交代
4. **独立检验**：第三方数据 + 与官方口径互相印证之处 + 独立测试发现的问题（尤其是官方没说的）
5. **价格与上手**：定价表（含上代与竞品对比）、入口清单、开源/发布节奏
6. **短板**：官方承认的已知限制原样列出，加使用建议
7. **来源**：按第 2 条格式收脚注；[END] 收束句在脚注之前

## 4. 发布类特有红线

- 未查证的媒体观点不转述（「据报道」也不行，除非自己打开过那篇报道）
- 厂商演示案例（如「48 小时设计芯片」）只有官方博客原文写了才可引用，且标注「官方演示，未经第三方复现」
- 幻觉率、安全性这类负面数据如有独立来源必须写，不许只报喜
- 时间敏感信息（价格、开源日期）写明「截至发稿」；发稿后重大变化应回改
- 配图只用官方博客发布的图或自绘示意图，禁止搬参考解读站的图（版权红线，同工具详情 v2 标准）

## 5. 与参考站（xiaohu 等）的关系

参考站只用来学**结构和呈现手法**（小节切法、口径披露方式、脚注格式），事实一律回到原始信源重新查证、文字全部重写。判断标准：把参考文章删掉，本文的每个数字仍能指出自己的出处。
