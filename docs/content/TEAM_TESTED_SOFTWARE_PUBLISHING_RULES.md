# 亲测文案与夸张营销边界规则 v1

Status: active

## 1. 核心规则

zhishare 的软件工具内容允许使用更强的真实写手风和营销表达。

可以写得尖锐、夸张、强营销、强痛点、强情绪、强反差。

但事实来源必须真实。

## 2. 三种体验来源

### A. 本人亲测

只有当用户明确说过自己真的测过时，才允许写：

- 本人亲测
- 我试过
- 我实际用下来
- 我踩过的坑
- 我测试后的结论

如果用户没有明确说“我试过”或等价表述，就不能把内容写成本人亲测。

### B. 团队亲测 / 站内实测

当用户明确说明软件已经由团队测试过，可以写：

- 团队亲测
- 站内实测
- 我们试过后收录
- 团队筛选后保留
- 站内测试后觉得值得关注

但具体测试细节仍然必须来自用户提供的信息。

如果用户没有给出具体过程，就不要编造：

- 测了几天
- 测了几个系统
- 测了哪些功能
- 遇到哪些 bug
- 跟哪些工具对比过
- 性能数据
- 价格变化
- 安全结论

### C. 网友评论摘选

可以搜索公开网友评论并做摘选或总结。

来源可以包括：

- GitHub issue
- GitHub discussion
- Reddit
- Hacker News
- Product Hunt
- X
- 应用商店评论
- 官方论坛
- 独立博客
- YouTube 评论
- B 站 / 小红书 / 知乎等公开讨论

可以写：

- 网友评论摘选
- 公共反馈摘选
- 社区反馈集中在
- GitHub issue 里有人提到
- Reddit 上有人讨论
- Product Hunt 评论里有人说
- 部分用户认为
- 也有用户吐槽

网友评论必须标注为公开反馈或公开评论摘选。

不能把网友评论包装成本人亲测。
不能把网友评论包装成团队亲测。
不能把网友评论包装成站内亲测。
不能把单条评论写成全网一致结论。
不能编造不存在的评论。

## 3. 允许的夸张表达

可以使用：

- 标题冲击力
- 强痛点描述
- 使用前后反差
- 小白焦虑
- 效率浪费
- 信息差
- 工具价值感
- 社媒开头钩子
- 结尾互动

示例：

- 这类工具最狠的地方，不是功能多，而是它直接打中了小白最痛的表达问题。
- 你以为自己缺的是工具，其实缺的是把需求说清楚的能力。
- 如果你还在手打一堆乱七八糟的需求，这个工具至少值得你看一眼。

## 4. 严禁编造的部分

禁止编造：

- 本人亲测
- 团队亲测
- 网友评论
- 用户数量
- 评分
- 下载量
- 价格
- 永久免费
- 许可证
- 安全审计
- 加密结论
- 隐私结论
- 平台支持
- 兼容性
- 性能数据
- 真实案例
- 客户反馈
- 和竞品对比结果

## 5. 工具库条目写法

工具库条目可以包含：

- 一句话爆点
- 团队亲测结论
- 官方功能介绍
- 适合谁
- 不适合谁
- 真实痛点
- 使用注意
- 风险提示
- 网友评论摘选
- 官方链接
- GitHub 链接
- 许可证
- 替代工具

工具库文案可以更强营销，但必须保留来源边界。

## 6. 文章写法

文章可以写成强真实写手风：

- 开头三行先给结论
- 先讲痛点
- 再讲为什么这个工具值得看
- 中间写亲测 / 团队亲测 / 网友评论
- 最后写适合谁和不适合谁
- 结尾留互动问题

如果是本人亲测文章：

- 可以写“我试过”
- 可以写“我踩坑”
- 可以写“实际用下来”

如果是团队亲测文章：

- 可以写“我们试过”
- 可以写“团队测试记录”
- 可以写“站内实测后收录”

如果是网友评论文章：

- 可以写“公开评论里有人提到”
- 可以写“社区反馈比较集中在”
- 不能写成站内测试

## 7. 官方来源核验仍然必须做

即使内容允许夸张，发布前仍要核验：

- 官网是否正确
- GitHub / GitLab / 官方仓库是否正确
- 文档是否来自官方
- 许可证是否清楚
- 下载入口是否官方
- 是否存在假官网、搬运站、破解版、盗版安装包
- 是否涉及侵权、灰产、去水印、绕过付费、隐私侵犯

如果官方来源不清楚，不能发布，只能进入候选池。

## 8. 禁止收录方向

以下内容不能收录：

- 破解软件
- 盗版下载
- 去水印工具
- 绕过付费工具
- 灰产引流工具
- 恶意采集工具
- 侵犯隐私工具
- 数据窃取工具
- 木马 / 后门 / 恶意脚本
- 来源不明安装包
- 只有网盘链接、没有官方来源的软件

## 9. 发布状态

建议使用以下状态：

### team_tested

用户或团队已测试，认为值得收录。

### copy_ready

ChatGPT 已编辑好工具页文案。

### ready_to_publish

Codex 可以写入数据库或内容文件，等待发布。

### published

已发布到网站。

### needs_more_info

缺少官网、链接、平台、价格、许可证、风险边界或测试说明。

### rejected

不适合收录。

## 10. Codex 的职责边界

Codex 只做：

- 写入文件
- 修改数据
- 运行 lint / build
- 提交
- 推送
- 按要求发布

Codex 不做：

- 代替本人测试
- 代替团队测试
- 代替站长判断
- 编造测试过程
- 编造网友评论
- 编造推荐理由
- 编造价格
- 编造许可证
- 编造截图
- 编造官网

## 11. ChatGPT 的职责边界

ChatGPT 负责：

- 提炼卖点
- 改写标题
- 写简介
- 写适合人群
- 写注意事项
- 写 SEO / GEO 摘要
- 写社媒文案
- 写工具详情页内容
- 搜索公开网友评论
- 摘选和总结公开反馈
- 根据用户提供的测试信息写真实写手文章

ChatGPT 不应该：

- 编造不存在的实测
- 编造不存在的网友评论
- 编造团队没有说过的结论
- 编造本人没有说过的体验
- 把官方宣传写成事实保证
- 把官方宣传包装成实测结论
- 把工具写成绝对安全
- 把风险工具包装成无风险工具
- 不能把官方宣传包装成实测结论

## 12. 后续标准流程

以后用户发来一个软件，流程改为：

1. 用户提供软件名称、官网、本人 / 团队测试结论。
2. ChatGPT 补充官方来源核验。
3. ChatGPT 可搜索网友评论摘选。
4. ChatGPT 生成工具库文案。
5. Codex 写入项目。
6. Codex lint / build。
7. Codex 提交推送。
8. 用户检查页面。
9. 如需深度文章，再由用户补充真实测试细节或允许使用公开评论摘选做素材。
