# 团队亲测与网友评论摘选发布规则 v1

Status: active

## 1. 核心规则

zhishare 的软件工具内容分工如下：

- 用户和团队负责筛选、试用、判断软件是否值得收录。
- 用户发给 ChatGPT 的软件，默认已经过用户或团队测试。
- ChatGPT 负责根据用户提供的信息和公开来源编辑文案。
- Codex 负责把文案写入项目、入库、提交和发布。
- Codex 不负责测试软件。
- Codex 不负责判断软件是否好用。
- Codex 不允许编造测试结果。
- Codex 不允许编造网友评论。

## 2. 允许的文案来源

### A. 团队亲测

用户明确发来的软件，可以默认按团队已筛选、团队已测试处理。

允许写：

- 团队亲测后收录
- 站内亲测
- 我们试过后觉得值得关注
- 团队筛选后保留
- 实际用下来适合某类用户
- 这类工具适合先收藏备用

注意：

具体体验细节仍然必须来自用户提供的信息。

如果用户没有说具体测试过程，就不要编造：

- 测了几天
- 测了几个系统
- 测了哪些功能
- 遇到哪些 bug
- 跟哪些工具对比过
- 性能数据
- 价格变化
- 安全结论

### B. 网友评论摘选

可以搜索公开网友评论并做摘选或总结。

来源可以包括：

- GitHub issue
- GitHub discussion
- Reddit
- Hacker News
- Product Hunt
- X
- 应用商店评论
- 官方论坛
- 独立博客
- YouTube 评论
- B 站 / 小红书 / 知乎等公开讨论

允许写：

- 网友反馈摘选
- 有人在公开评论里提到
- 社区反馈集中在
- GitHub issue 里有人反映
- Reddit 上有人讨论
- Product Hunt 评论里有人说
- 部分用户认为
- 也有用户吐槽
- 网友评论必须标注为公开反馈或公开评论摘选

禁止写：

- 全网一致好评
- 所有人都说好
- 口碑炸裂
- 没有差评
- 用户都觉得完美
- 编造不存在的评论
- 把网友评论写成团队亲测
- 把单条评论写成普遍结论

## 3. 工具库条目写法

工具库条目可以包含：

- 工具名称
- 一句话简介
- 团队亲测结论
- 适合谁
- 不适合谁
- 功能亮点
- 使用注意
- 权限 / 数据 / 平台边界
- 官方网站
- 官方仓库
- 许可证
- 网友评论摘选
- 替代工具

工具页可以写得更像运营文案，但仍要避免虚假结论。

## 4. 真写手文章写法

真写实测文章可以写：

- 我试过
- 我踩坑
- 实际用下来
- 翻车点
- 真实观察
- 最后怎么选
- 团队测试记录
- 网友评论摘选

但要求：

- 亲测内容来自用户或团队
- 网友评论来自公开来源
- 评论摘选要标注来源
- 不能把官方宣传当作亲测
- 不能把网友评论当作站内测试
- 不能把网友评论包装成站内亲测
- 不能编造时间、设备、数据、结果

## 5. 网友评论摘选格式

建议统一使用：

```text
网友反馈摘选：
- GitHub issue：有人提到……
- Reddit：有人讨论……
- Product Hunt：评论集中在……
- 应用商店：部分用户反馈……
```

如果只找到少量评论，写：

```text
目前公开评论不多，只能作为参考，不能代表整体口碑。
```

如果评论分歧明显，写：

```text
公开反馈有分歧：有人觉得它解决了痛点，也有人认为……
```

## 6. 官方来源核验仍然必须做

即使团队已测，发布前仍要核验：

- 官方网站是否正确
- GitHub / GitLab / 官方仓库是否正确
- 文档是否来自官方
- 许可证是否清楚
- 下载入口是否官方
- 是否存在假官网、搬运站、破解版、盗版安装包
- 是否涉及侵权、灰产、去水印、绕过付费、隐私侵犯

如果官方来源不清楚，不能发布，只能进入候选池。

## 7. 禁止收录方向

以下内容不能收录：

- 破解软件
- 盗版下载
- 去水印工具
- 绕过付费工具
- 灰产引流工具
- 恶意采集工具
- 侵犯隐私工具
- 数据窃取工具
- 木马 / 后门 / 恶意脚本
- 来源不明安装包
- 只有网盘链接、没有官方来源的软件

## 8. 发布状态

建议使用以下状态：

### team_tested

用户或团队已测试，认为值得收录。

### copy_ready

ChatGPT 已编辑好工具页文案。

### ready_to_publish

Codex 可以写入数据库或内容文件，等待发布。

### published

已发布到网站。

### needs_more_info

缺少官网、链接、平台、价格、许可证、风险边界或测试说明。

### rejected

不适合收录。

## 9. Codex 的职责边界

Codex 只做：

- 写入文件
- 修改数据
- 运行 lint / build
- 提交
- 推送
- 按要求发布

Codex 不做：

- 代替团队测试
- 代替站长判断
- 编造测试过程
- 编造网友评论
- 编造推荐理由
- 编造价格
- 编造许可证
- 编造截图
- 编造官网

## 10. ChatGPT 的职责边界

ChatGPT 负责：

- 提炼卖点
- 改写标题
- 写简介
- 写适合人群
- 写注意事项
- 写 SEO / GEO 摘要
- 写社媒文案
- 写工具详情页内容
- 搜索公开网友评论
- 摘选和总结公开反馈
- 根据用户提供的测试信息写真实写手文章

ChatGPT 不应该：

- 编造不存在的实测
- 编造不存在的网友评论
- 编造团队没有说过的结论
- 把官方宣传写成事实保证
- 把工具写成绝对安全
- 把风险工具包装成无风险工具

## 11. 后续标准流程

以后用户发来一个软件，流程改为：

1. 用户提供软件名称、官网、团队测试结论。
2. ChatGPT 补充官方来源核验。
3. ChatGPT 可搜索网友评论摘选。
4. ChatGPT 生成工具库文案。
5. Codex 写入项目。
6. Codex lint / build。
7. Codex 提交推送。
8. 用户检查页面。
9. 如需深度文章，再由用户补充真实测试细节或允许使用公开评论摘选做素材。
