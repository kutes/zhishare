## 开头结论

我先把结论放在前面：Privacy Filter 这个方向是对的，但我现在还不能把它写成“推荐”。
这不是推荐稿。

它的官方页面看起来像一个认真解决问题的工具：在浏览器里检测姓名、邮箱、电话、地址、账号、日期、 URL 和密钥，一键生成脱敏文本。

这听起来很有用，但我这次做的是一次真实观察，不是写推荐语。

我在本地浏览器里跑了两轮观察，看到的结果比较克制：

- 页面能打开。
- 模型和 tokenizer 相关资源会下载。
- 我没有看到虚构样本通过 POST / PUT / PATCH 上报。
- 但两轮里都没有拿到稳定的最终脱敏输出。
- 界面主要卡在加载态。

所以这篇不是“工具评测推荐”，而是一次真实的观察记录。

## 我为什么会去测它

我现在越来越不敢直接把原始资料粘给 AI。

不是因为 AI 一定会出问题，而是因为很多人的流程太顺手了：

```text
客户资料 -> 复制 -> 粘贴到 AI -> 让 AI 整理
```

这个流程太顺了，顺到你可能根本没注意里面其实有：

- 姓名
- 手机号
- 邮箱
- 地址
- 订单号
- API Key
- 数据库密码
- 公司内部备注

等你意识到的时候，内容可能已经发出去了。

所以我真正想测的不是“Privacy Filter 是什么”，而是：

```text
在我把资料发给 AI 之前，它能不能先帮我挡一下手滑？
```

## 我这次用的样本

我没有用真实客户资料，也没有用真实密钥。

我用的是一段虚构样本：

```text
客户姓名：张明
手机号码：13812345678
邮箱：zhangming@example.com
地址：北京市朝阳区测试路 88 号
订单号：ORDER-2026-0001
API Key：sk-test-1234567890abcdef
数据库密码：Test_DB_Password_2026!
备注：客户希望下周三前收到合同草稿，并要求不要公开联系方式。
```

这段样本的目的很简单：看它能不能识别几类常见敏感信息：姓名、手机号、邮箱、地址、订单号和各种格式的 API Key。

## 官方页面怎么说

Privacy Filter 官方页面的说法是：

- 它可以检测姓名、邮箱、电话、地址、账号、日期、 URL 和密钥。
- 它可以生成脱敏文本。
- 它基于 openai/privacy-filter 模型。
- 首次运行会从 Hugging Face 下载模型文件。
- 推理通过 Transformers.js 在浏览器里完成。
- 官方页面声称文本不会上传到服务器。

## 我第一次和第二次观察到什么

我做了两轮观察，情况基本一致：

- 页面可以打开。
- 能看到输入框。
- 能看到模型 / tokenizer / 静态资源下载。
- 我没有看到虚构样本进入 POST / PUT / PATCH 请求体。
- 但是两轮都没有稳定拿到最终脱敏输出。
- 页面主要卡在加载态。

所以现在这个段落，我只能写：

```text
它的方向是对的，但我现在还不能把它写成推荐。
```

## 我现在怎么判断它

我现在只能给出一个中间判断：

- 方向值得关注。
- 页面设计理念是对的。
- 本地处理这个思路也很重要。
- 但在我这两轮观察里，它还没有稳定给出最终脱敏输出。

所以我不会把它写成“推荐大家马上用”的那种文章。

我更愿意把它写成一个值得继续观察的本地脱敏工具。

方向值得关注，但我现在还不能把它写成推荐。

## 我给普通人的建议

如果你平时经常把客户资料、工单、日志、聊天记录发给 AI，我建议你先记住一件最简单的事：

```text
上传之前，先手动删掉姓名、电话、邮箱、地址、密钥和客户资料。
```

工具可以帮你多一道检查，但它不该替你承担全部责任。
它不能替代人工审核。

## 最后

这次观察没有把 Privacy Filter 写成一篇推荐稿。

我更想把它留在一个更真实的位置上：

- 它可能有用。
- 它的方向值得关注。
- 但我在本地两轮观察里还没有得到稳定最终脱敏输出。
- 所以现在不能下结论说它已经够稳。

这篇文章后面如果继续写，我会优先补三件事：

1. 换更多样本。
2. 换不同浏览器复测。
3. 等模型完整缓存后再测一次。
