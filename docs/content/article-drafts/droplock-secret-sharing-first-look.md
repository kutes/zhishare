# 我以后不敢直接微信发 API Key 了：这种临时加密分享工具靠谱吗？

## 开头结论

先把结论放前面：DropLock 这个方向值得继续关注，但它不是“安全神话”。

我这次不是在写结论稿，而是在写一篇真实观察。
我用一段虚构的 secret 跑了一遍流程，结果比我预期顺，但也没有顺到可以直接下最终结论。

它的官方页面看起来像一个专门解决“临时 secret 怎么传”这个问题的工具：在浏览器里输入内容，生成 lock box link，再把 secret 锁进一个带 `#m=` fragment 的链接里。
从体验上看，这比把 API Key 直接丢进微信、QQ、Telegram 或邮件里要更克制一些。

但我必须先说清楚：

- 这不是安全背书；
- 这不是“绝对安全”；
- 这也不是企业级密钥管理的替代品；
- 官方页面明确提醒，它还没有经过安全专家审计。

所以，这篇文章更像一篇现场观察笔记。

## 我为什么会去测它

很多人传 secret 的方式，其实非常随手。

你可能也见过这些场景：

- “我把 API Key 发微信了。”
- “密码在上面那个群里。”
- “临时邀请码我发邮件给你。”

问题不在于你故意不安全，而在于这些动作太顺手了。
一旦 secret 进入聊天记录，它就很难再回到你能完全掌控的位置：

- 聊天记录会长期保存；
- 截图容易被转发；
- 对方复制后你很难追踪；
- 轮换的时候经常没人记得；
- 真出问题后，排查也很痛苦。

所以我真正想看的不是“DropLock 是什么”，而是：

```text
它能不能让临时传 secret 这件事，比直接扔聊天框更可控一点？
```

## 我这次用的样本

我没有用真实 API Key，也没有用真实数据库密码。

我用的是一段虚构样本：

```text
DEMO_OPENAI_KEY=sk-test-droplock-demo-1234567890
DEMO_DB_PASSWORD=Demo_DB_Password_2026!
DEMO_INVITE_CODE=ZHISHARE-DEMO-2026
```

这只是为了观察流程，不代表任何真实账户、真实系统或真实权限。

## 官方页面怎么说

官方页面对它的描述，大概是这样的：

- 可以在浏览器里检测和处理 secret；
- 可以生成 lock box link；
- 接收方打开后再输入 secret；
- 生成带 `#m=` fragment 的 secret link；
- secret 不通过普通表单请求直接上传；
- 主要处理逻辑是在浏览器侧完成；
- 页面也明确提示没有经过安全专家审计。

这个叙事很清楚：它想解决的是“临时传递敏感文本”这件事。

但我要强调，这种叙事和“安全已验证”不是一回事。

## 我观察到的流程

我跑了两轮观察，流程基本一致：

1. 页面可以打开；
2. 能看到输入框；
3. 模型 / tokenizer / 静态资源会下载；
4. 我没有看到虚构样本通过 POST / PUT / PATCH 进入请求体；
5. 生成链接时，页面会生成带 `#m=` 的 secret link；
6. 重新打开这个链接后，页面会回到接收侧状态；
7. 我没有观察到服务端删除接口调用；
8. 主要卡点还是加载阶段。

这里最值得记录的是最后两点：

- 它确实在尝试把 secret 留在 fragment 层；
- 但我还没有拿到稳定、可复核的最终脱敏输出结果。

所以我不会把它说成“已经稳了”。

## 网络请求里看到了什么

我重点看了网络层。

目前我看到的主要是：

- GET 文档；
- GET logo.png；
- 没有观察到 POST；
- 没有观察到 PUT；
- 没有观察到 PATCH；
- 没有观察到虚构 secret 作为明文进入请求体。

这和它“把结果放在 URL fragment”这个方向是对得上的。

但我还是要补一句：

```text
我没有观察到虚构 secret 通过 POST / PUT / PATCH 请求体发送，
不等于它已经通过了安全审计。
```

这只是一次本地观察，不是安全证明。

## 它和普通聊天框最大的区别

普通聊天框最大的问题，不只是“你发出去了”。

而是你发出去以后，secret 很容易变成：

- 聊天记录的一部分；
- 多人可见的复制文本；
- 截图里的固定痕迹；
- 很难主动撤回的内容。

DropLock 这类工具的思路不一样。

它试图先生成一个 lock box link，再把 secret 锁进去，最后返回一个带 `#m=` 的链接。
从体验上，这比“直接扔聊天框”更克制一点。

但这不代表它能替代：

- 1Password；
- Bitwarden；
- Vault；
- 云厂商 Secret Manager；
- 企业权限和审计流程。

它更像是一个“临时短文本怎么传”的工具，不是完整的 secret 管理体系。

## 我现在能确认什么

目前我能比较稳地说：

- 官方页面能打开；
- 不需要注册账号；
- 能生成 lock box link；
- 能输入虚构 secret；
- 能生成带 `#m=` 的 secret link；
- 网络观察里没有看到虚构 secret 通过 POST / PUT / PATCH 请求体发送；
- 官方页面明确提示它没有经过安全专家审计。

这些足够让我写一篇观察稿，但还不足以让我下“可以放心大规模使用”的结论。

## 我现在不能确认什么

我现在不能确认：

- 加密实现是否没有边界缺陷；
- 旧链接和浏览器历史记录会带来什么风险；
- 不同浏览器 profile 的行为是否一致；
- 长文本、大 secret、特殊字符是否都稳定；
- 手机浏览器体验是否可靠；
- 是否适合团队生产流程；
- 是否适合传真实生产密钥。

这些都需要继续复测和代码复核。

## 风险点

我最在意的第一个风险点，是链接替换。

如果 lock box link 在传输或复制过程中被替换，发送方最后可能把 secret 锁给了错误的人。
这不是小事。

第二个风险点，是浏览器 profile。

secret 可能和当前浏览器 profile 绑定。
换设备、换浏览器、清缓存、换 profile，都可能影响读取。

第三个风险点，是误以为它能替代企业级密钥管理。它不应替代企业级密钥管理。

它不应该替代：

- 生产 API Key；
- 数据库 root 密码；
- SSH 私钥；
- 公司内部核心凭证；
- 需要审计、权限和撤回机制的团队密钥流程。

如果你真正要管的是生产级 secret，还是应该走正式的 secret management。

## 谁适合继续关注

我觉得下面这些人适合继续关注它：

- 独立开发者；
- 小团队；
- 临时协作的人；
- 经常传 API Key、邀请码、临时密码的人；
- 想减少聊天记录里 secret 痕迹的人；
- 愿意理解 basic tradeoff 的用户。

但这不等于我建议你马上把它塞进生产环境。

## 谁不适合直接依赖它

下面这些场景，不适合直接依赖它：

- 生产环境 API Key；
- 数据库 root 密码；
- 服务器 SSH 私钥；
- 客户数据；
- 公司内部核心凭证；
- 长期团队密钥管理；
- 需要审计、权限和撤回机制的正式流程。

这些应该走正经的 secret management。

## 如果你也常在聊天框里发 secret

我给普通人的建议其实很简单：

```text
发出去之前，先手动删掉姓名、电话、邮箱、地址、密码和客户资料。
```

工具可以帮你多检查一层，但它不能替你负责全部后果。

如果你经常传 secret，真正值得养成的习惯不是“找一个看起来很酷的工具”，而是：

- 先确认这个 secret 到底有没有必要发；
- 能临时就别永久；
- 能轮换就轮换；
- 能用正式 secret manager 就别靠聊天记录。

## 结尾

这次观察让我觉得，DropLock 的方向是有意思的。

它至少提醒了我一件事：我现在传的不是普通文字，而是 secret。

但我现在还不能把它写成推荐。

更准确的说法是：

它值得继续观察，但还不应该被写成“已经足够可靠”的结论。

如果后面继续写，我会优先补三件事：

1. 换更多样本；
2. 换不同浏览器复测；
3. 等模型完整缓存后再测一次。
