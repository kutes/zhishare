# Privacy Filter 二创实测任务单 v1

## 1. 资源定位

资源名：

Privacy Filter

文章方向：

我把客户资料发给 AI 前，先用本地脱敏工具清了一遍

备用标题：

发给 AI 前先脱敏：这种隐私过滤工具真能防手滑吗？

这不是一篇“Privacy Filter 是什么”的百科文。

这篇文章要回答的是：

- 小白把资料发给 AI 前，最容易泄露什么？
- 本地脱敏工具能不能识别常见敏感信息？
- 它会漏掉哪些信息？
- 它会不会误伤正常内容？
- 它是否真的本地处理？
- 它适合谁？
- 它不适合谁？
- 它能不能替代人工检查？

## 2. 为什么先测它

Privacy Filter 适合优先二创，原因是：

- 它和“免费 AI 工具安全吗”这个主题天然相连
- 它解决的是发给 AI 前的隐私风险
- 它适合做低敏样本实测
- 它能写出小白能懂的避坑内容
- 它不涉及侵权下载、破解或灰产方向

## 3. 测试前统一规则

测试时禁止使用真实敏感资料：

- 真实客户姓名
- 真实电话号码
- 真实邮箱
- 真实住址
- 真实订单
- 真实身份证
- 真实公司资料
- 真实聊天记录
- 真实 API Key
- 真实数据库密码
- 真实未公开代码

测试只使用虚构样本。

## 4. 低敏测试样本

统一使用这段虚构文本：

```text
客户姓名：张明
手机号码：13812345678
邮箱：zhangming@example.com
地址：北京市朝阳区测试路 88 号
订单号：ORDER-2026-0001
API Key：sk-test-1234567890abcdef
数据库密码：Test_DB_Password_2026!
备注：客户希望下周三前收到合同草稿，并要求不要公开联系方式。
```

注意：

这段内容全部是虚构样本，不能替换成真实资料。

## 5. 核心测试任务

### 任务 A：基础 PII 脱敏

把虚构样本输入 Privacy Filter。

观察：

- 姓名是否被识别
- 手机号是否被识别
- 邮箱是否被识别
- 地址是否被识别
- 订单号是否被识别

### 任务 B：密钥和密码脱敏

观察：

- API Key 是否被识别
- 数据库密码是否被识别
- 是否只识别 `sk-` 开头的 key
- 是否能识别普通格式密码

### 任务 C：上下文保留

观察：

- 脱敏后句子是否还能读懂
- 是否保留任务背景
- 是否影响后续 AI 理解
- 是否把太多正常内容也遮掉

### 任务 D：漏检与误伤

观察：

- 有没有明显漏掉的敏感字段
- 有没有误伤普通词
- 有没有把“合同草稿”等非敏感内容也误遮住

### 任务 E：本地处理核验

观察：

- 工具是否明确说明本地处理
- 浏览器网络面板是否有上传请求
- 官方文档是否说明数据不会离开本地
- 是否依赖第三方 API

## 6. 文章要回答的问题

完成测试后，文章必须回答：

- 它能不能减少发给 AI 前的隐私风险？
- 哪些字段识别比较稳？
- 哪些字段容易漏掉？
- 它会不会误伤正常内容？
- 它能不能替代人工检查？
- 小白是否容易用错？
- 适合什么人？
- 不适合什么人？
- 如果只给新手一句建议，会是什么？

## 7. 文章开头预案

未完成测试前只允许这样写：

```text
我先说结论：把客户资料、日志、合同草稿直接发给 AI，是很多人最容易忽略的隐私风险。Privacy Filter 这类工具的价值，不是让你“绝对安全”，而是在发送前帮你多做一层脱敏检查。当前这篇还处在任务单阶段，我会用一段虚构客户资料测试它能识别哪些敏感信息、会漏掉哪些内容，以及它能不能替代人工检查。
```

## 8. 发布前必须补齐

发布前必须补齐：

- 官方来源核验
- 是否开源
- 是否本地处理
- 是否有上传请求
- 许可证或使用条款
- 5 个测试任务观察结果
- 至少 3 个真实观察结论
- 至少 3 个新手避坑建议
- 至少 1 个不推荐场景
- 最终判断
- 互动问题

## 9. 禁止发布条件

如果以下任一项未完成，不得发布：

- 没有真实观察表
- 没有官方来源核验
- 没有本地处理核验
- 没有隐私风险说明
- 没有不适合人群
- 没有风险提醒
- 只有泛泛而谈
- 看起来像 AI 生成百科
