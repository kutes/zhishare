## 3. 为什么测它

很多小白不是不会用 AI，而是一开始就说不清楚：

```text
帮我写一个网站文案，要高级一点，别太长，也别太普通，最好能吸引人。
```

这类需求交给 AI，结果通常也很普通。

OpenLess 值得测，是因为它瞄准的是 AI 使用前的第一步：

```text
把人嘴里混乱的想法，整理成可用的 Prompt。
```

这和 zhishare 的真实写手风格很贴合。

重点不是吹它多强，而是测：

- 它能不能理解混乱口述？
- 它能不能保留限制条件？
- 它会不会擅自扩写？
- 它生成的 Prompt 是否真的更适合发给 AI？
- 它的权限和本地优先边界是否说得清楚？

## 5. 低敏测试样本

统一使用 3 段虚构口述任务。

### 样本 A：网站改版需求

```text
我想让 AI 帮我改一下网站首页，黑色风格，高级一点，不要太花，标题要简短，然后工具卡片别像普通列表，最好让人觉得这是一个很认真筛选资源的网站，移动端也要好看。
```

### 样本 B：邮件回复需求

```text
帮我回一个客户邮件，客户说报价有点高，我想表达我们可以微调方案，但不能直接大幅降价，语气要礼貌一点，不要太硬，也别显得我们很着急成交。
```

### 样本 C：文章选题需求

```text
我想写一篇文章，主题是发给 AI 前先脱敏，不要写成工具推荐，要写真实一点，最好说清楚我测试时遇到什么问题，还有哪些地方不能下结论。
```

## 6. 核心测试任务

### 任务 A：基础口述转 Prompt

观察：

- 能否按热键开始
- 能否识别口述
- 能否输出文本
- 能否把废话整理成结构
- 能否保留原始需求
- 能否乱加不存在的信息

### 任务 B：AI Prompt 场景

观察：

- 输出是否适合直接发给 ChatGPT / Claude / Cursor
- 是否包含目标、约束、风格、输出格式
- 是否保留“不要做什么”
- 是否比原始口述更清楚

### 任务 C：任意输入框插入

观察：

- ChatGPT 输入框是否可插入
- 浏览器文本框是否可插入
- 记事本 / 文本编辑器是否可插入
- 如果不能插入，是否会复制到剪贴板
- 光标位置是否准确

### 任务 D：权限边界

观察：

- 是否需要麦克风权限
- 是否需要辅助功能权限
- 是否需要重启应用后权限才生效
- 用户是否容易理解这些权限
- 权限提示是否清楚

### 任务 E：本地优先与网络边界

观察：

- 音频、转写、词典、历史是否有本地存储说明
- 是否需要配置云 ASR 或 LLM 提供商
- 是否需要 API 凭证
- 网络请求里是否能看到外部服务
- 本地优先是否等于完全离线
- 用户是否容易误解
