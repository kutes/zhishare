# ARTICLE_CONTENT_STANDARD

## 用途

文章内容与排版的统一标准。2026-07-10 制定,来源是对 [best.xiaohu.ai](https://best.xiaohu.ai/) 文章详情页(实测样本:`/article/seedream-5-pro-launch/`)的结构化分析。**以后所有文章的撰写与渲染都按本文件执行**,不要每次重新设计。

与 `docs/DESIGN_SYSTEM.md` 的关系:DESIGN_SYSTEM.md 管颜色/字体/卡片这些跨页面的视觉令牌;本文件管**文章这一种内容类型**的写作结构与专属渲染组件。

## 研究来源与方法论

直接 `curl` 抓取目标页面 HTML(非视觉截图),逐段解析真实标记结构,而非凭观感臆测。以下每条规则后标注了原始证据。

## 核心标准(必须遵守)

### 1. 开篇来源披露

文章正文第一段之前,用一句**弱化字号**的文字说明信息来源与性质。

原始范例:
> 本文基于字节跳动 Seed 团队官方发布内容整理，文中能力说明、示例图片与视频均来自厂商，属官方演示效果，未经第三方独立复现。

zhishare 适配:每篇文章正文开头(在 sections 之前)加一行**来源说明**,固定句式:

```
本文基于 {信息来源} 整理，{内容性质说明（如：属官方演示效果/为个人测试体验/为公开资料汇总）}。
```

这与首页 Hero 的"来源清晰、优先官网"口号是同一件事的落地,**不是可选项**。

### 2. 编号小节(取代扁平 h2 列表)

每个内容小节 = 一个带编号的"章"，不是简单的标题+段落。

原始标记:
```html
<div class="sec first hook">
  <div class="eyebrow"><span class="n">1</span> 发布</div>
  <h2>不止会"画"，更懂"排版"</h2>
  ...
</div>
```

结构规则:
- 小节按顺序编号(1、2、3...),编号 + 一个 2-4 字的短标签(如"发布""真实感""语言")放在标题上方一行,弱化处理。
- `<h2>` 本身要有**画面感或悬念**,不是干巴巴的功能名(反例:"功能介绍";正例:"文字说不清改哪儿，模型现在看得懂坐标")。
- 全文小节数量参考:一篇中等深度文章 5-9 个小节。zhishare 文章更短,3-6 个小节即可,不必凑数。

### 3. 双段式导语(每个小节开头)

不是"标题下面直接堆一段话",而是**结论段 + 展开段**两段,权重不同。

原始标记:
```html
<p class="lead">字节跳动 Seed 团队今天（7 月 8 日）正式发布多模态图像创作模型 Seedream 5.0 Pro，已上线火山方舟体验中心，接下来会陆续接入豆包和即梦。</p>
<p class="big">相比上一代，它在图文匹配、结构合理、文字渲染、画面美感这些基础能力上做了整体升级，同时带来四个方向的新能力：...</p>
```

写作规则:
- **第一段（lead）**:先说结论——是什么、谁做的、什么时候。字号最大,一句话说完事实。
- **第二段（big）**:展开——这件事具体包含哪几点、和上一版/同类比有什么不同。字号次大,仍比正文重。
- 不是每个小节都要两段都有,但开篇小节必须有。

### 4. "为什么值得看" 框(全文只用一次,放在开篇)

原始标记:
```html
<div class="why">
  <span class="ic">◆</span>
  <div><b>为什么值得看：</b>信息图生成被官方称为当前 AI 图像生成里复杂度最高的领域之一……</div>
</div>
```

zhishare 适配:每篇文章开篇小节里放一个**"为什么值得关注"框**,用编辑视角回答"这事和读者有什么关系",不是简单复述新闻本身。全文只出现一次,是给这篇文章定调的地方。

### 5. 关键结论高亮框(每个重要转折处)

原始标记:
```html
<div class="keypoint">
  <span class="tag">核心能力 · 三</span>
  <p>一张海报能被拆成文字、主体、背景、环境装饰等<b>十余个独立图层</b>……</p>
</div>
```

写作规则:
- 一篇文章遇到"这是本文的一个核心卖点/结论"时,单独抽出来做一个高亮框,不要埋在大段落里。
- 框内有一个短标签(如"核心能力 · 三"、"避坑点 · 二"),正文里加粗最关键的短语（如"十余个独立图层"）。
- 一篇文章通常 2-4 个这样的框,数量对应文章的核心论点数,不是装饰,是内容大纲的外化。

### 6. 图片块与视频块(都必须标来源)

原始标记(图片):
```html
<div class="photo-block reveal">
  <img loading="lazy" src="..." alt="南极秦岭站科考信息图，含时间轴、柱状图、饼图、折线图与站体实景">
  <div class="photo-cap">「南极秦岭站」信息图：一张图里排下科考时间轴……来源：字节跳动Seed</div>
</div>
```

原始标记(视频,复用同一套 caption 样式):
```html
<div class="video-block reveal">
  <video controls preload="metadata" playsinline src="..."></video>
  <div class="photo-cap">演示：整张海报拆成十余个透明图层……来源：字节跳动Seed</div>
</div>
```

写作规则:
- **图片/视频说明文字必须以"来源：XXX"结尾**——这一条零容忍,不标来源不发。
- `alt` 属性写实际描述(无障碍 + SEO),不是留空。
- 这一节的机制在 zhishare 已经用工具详情页的 `ToolMediaGallery`(2026-07-09 建的)实现过一次,文章详情页复用同一套组件与存储模式(`article-media/{slug}.json`,不是新发明)。

### 7. 相关推荐(文末)

原始标记:每张卡一个 `--tint` 强调色、分类角标、标题、日期·阅读时长。

zhishare 现状:`article-detail-related-card` 已经有类似结构(分类 tag + 标题 + 摘要 + 链接),**不需要重做**,只需确认视觉语言与本标准的其余部分统一(暖色调、衬线标题)。

## 可选组件(不强制,按内容需要使用)

以下三个来自 xiaohu 但**不适合 zhishare 内容形态**,降级为可选,不是标准的必修部分:

| 组件 | xiaohu 原用途 | zhishare 何时用 |
| --- | --- | --- |
| 代码/参数示例块(原 `.snippet`,`<pre>` + 标签) | 展示 AI 生图的完整 Prompt | 只有涉及命令行、配置代码、脚本片段的文章才用(如"自动化脚本"类工具教程) |
| 多图对比网格(原 `.imgrid`/`.fourgrid`) | 一次性铺开多张生成结果对比 | 只有需要并排比较多张截图时才用(如"5 个工具对比"类文章) |
| 横向滚动案例墙(原 `.showcase`,文末链官方案例页) | 引导去官方产品页看更多案例 | 只有被报道对象自己有丰富官方案例库时才用,大多数 zhishare 文章不需要 |

## 数据模型影响(仅记录,不在本文件实施)

当前 `ArticleSection` 类型(`src/components/articles/article-content.ts`)只有 `paragraphs` 和 `list` 两种,不足以承载上述结构。落地本标准需要扩展该类型并改造 `parseArticleSections`/`ArticleContentSection` 渲染逻辑,新增:

- 小节级 `lead`/`big` 双段导语字段
- `why` 类型(全文一次)
- `keypoint` 类型(tag + 加粗正文)
- `photo`/`video` 类型(复用 `ToolMediaGallery` 的存储与白名单嵌入模式,重命名为通用的 `ArticleMediaGallery` 或直接扩展现有组件)
- 可选:`snippet`(代码块)、`imgrid`(多图网格)、`showcase`(横向滚动)

这是一次真正的功能开发(类型改造 + 内容解析 + 渲染组件 + 至少一篇真实文章按新结构重写做验证),按项目惯例应该走 `brainstorming → 设计确认 → 写 spec → 写 plan → 执行` 的完整流程,是下一步的独立任务,不在本次研究范围内直接动代码。

## 执行检查清单(未来写/改任何文章前过一遍)

- [ ] 每个小节标题前有编号 + 短标签
- [ ] 开篇小节是 lead + big 双段
- [ ] 有且只有一个"为什么值得看"框,在开篇
- [ ] 每个核心论点都有对应的高亮框(2-4 个)
- [ ] 每张配图/视频的说明文字以"来源：XXX"结尾
- [ ] 相关推荐区展示同分类文章
