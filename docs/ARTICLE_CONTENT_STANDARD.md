# ARTICLE_CONTENT_STANDARD

## 用途

文章内容与排版的统一标准。2026-07-10 制定,来源是对 [best.xiaohu.ai](https://best.xiaohu.ai/) 文章详情页(实测样本:`/article/seedream-5-pro-launch/`)的结构化分析;2026-07-16 用第二个样本(`/article/thinking-machines-inkling-open-model/`)复核并补充(见「v2 补充」一节)。**以后所有文章的撰写与渲染都按本文件执行**,不要每次重新设计。

与 `docs/DESIGN_SYSTEM.md` 的关系:DESIGN_SYSTEM.md 管颜色/字体/卡片这些跨页面的视觉令牌;本文件管**文章这一种内容类型**的写作结构与专属渲染组件。

## 研究来源与方法论

直接 `curl` 抓取目标页面 HTML(非视觉截图),逐段解析真实标记结构,而非凭观感臆测。以下每条规则后标注了原始证据。

## 核心标准(必须遵守)

### 0. 开篇「一分钟速览」(2026-07-10 补充,首版遗漏)

正文第一节之前,放一个**要点速览框**:小标题「一分钟速览」+ 3-5 条要点列表,读者不读全文也能带走核心信息。

原始标记(位于副标之后、来源披露之前):
```html
<div class="tldr">
  <div class="h">一分钟速览</div>
  <ul>
    <li>字节跳动 Seed 团队发布多模态图像创作模型 Seedream 5.0 Pro……</li>
    <li>新增复杂信息可视化：一张图里同时融合时间轴、图表……</li>
    ...
  </ul>
</div>
```

写作规则:每条要点是一个完整句(有主语有结论),不是名词短语;3-5 条,对应文章核心论点;**顺序 = 全文小节顺序**,速览就是文章的目录式浓缩。

### 1. 开篇来源披露

文章正文第一段之前,用一句**弱化字号**的文字说明信息来源与性质。

原始范例:
> 本文基于字节跳动 Seed 团队官方发布内容整理，文中能力说明、示例图片与视频均来自厂商，属官方演示效果，未经第三方独立复现。

zhishare 适配:每篇文章正文开头(**速览框之后、第一节之前**——与目标站顺序一致)加一行**来源说明**,前缀 ⚑ 旗标,固定句式:

```
⚑ 本文基于 {信息来源} 整理，{内容性质说明（如：属官方演示效果/为个人测试体验/为公开资料汇总）}。
```

原始佐证:目标站的披露是 `<div class="stance"><span>⚑</span><span>本文基于……</span></div>`,位于 `.tldr` 速览框之后、`.sec first hook` 第一节之前,是一个体系化的"立场透明"装置(他们的首页卡片上也有对应的 `hb-stance` 角标)。

这与首页 Hero 的"来源清晰、优先官网"口号是同一件事的落地,**不是可选项**。

### 2. 编号小节(取代扁平 h2 列表)

每个内容小节 = 一个带编号的"章"，不是简单的标题+段落。

原始标记:
```html
<div class="sec first hook">
  <div class="eyebrow"><span class="n">1</span> 发布</div>
  <h2>不止会"画"，更懂"排版"</h2>
  ...
</div>
```

结构规则:
- 小节按顺序编号(1、2、3...),编号 + 一个 2-4 字的短标签(如"发布""真实感""语言")放在标题上方一行,弱化处理。
- `<h2>` 本身要有**画面感或悬念**,不是干巴巴的功能名(反例:"功能介绍";正例:"文字说不清改哪儿，模型现在看得懂坐标")。
- 全文小节数量参考:一篇中等深度文章 5-9 个小节。zhishare 文章更短,3-6 个小节即可,不必凑数。

### 3. 双段式导语(每个小节开头)

不是"标题下面直接堆一段话",而是**结论段 + 展开段**两段,权重不同。

原始标记:
```html
<p class="lead">字节跳动 Seed 团队今天（7 月 8 日）正式发布多模态图像创作模型 Seedream 5.0 Pro，已上线火山方舟体验中心，接下来会陆续接入豆包和即梦。</p>
<p class="big">相比上一代，它在图文匹配、结构合理、文字渲染、画面美感这些基础能力上做了整体升级，同时带来四个方向的新能力：...</p>
```

写作规则(2026-07-10 修正:经实测核对,首版对 big 的适用范围描述有误):
- **lead(每个小节都有)**:每节第一段都是 lead——先说结论,字号最大。实测目标站 9 个小节 9 个 lead,无一例外。
- **big(全文只有一处,在开篇节)**:开篇节的第二段用 big 展开——具体包含哪几点、和同类比有什么不同。实测全文仅 1 个 big。**其它小节的第二段就是普通正文,不要放大**,否则全文视觉过重。

### 4. "为什么值得看" 框(全文只用一次,放在开篇)

原始标记:
```html
<div class="why">
  <span class="ic">◆</span>
  <div><b>为什么值得看：</b>信息图生成被官方称为当前 AI 图像生成里复杂度最高的领域之一……</div>
</div>
```

zhishare 适配:每篇文章开篇小节里放一个**"为什么值得关注"框**,用编辑视角回答"这事和读者有什么关系",不是简单复述新闻本身。全文只出现一次,是给这篇文章定调的地方。

### 5. 关键结论高亮框(每个重要转折处)

原始标记:
```html
<div class="keypoint">
  <span class="tag">核心能力 · 三</span>
  <p>一张海报能被拆成文字、主体、背景、环境装饰等<b>十余个独立图层</b>……</p>
</div>
```

写作规则:
- 一篇文章遇到"这是本文的一个核心卖点/结论"时,单独抽出来做一个高亮框,不要埋在大段落里。
- 框内有一个短标签(如"核心能力 · 三"、"避坑点 · 二"),正文里加粗最关键的短语（如"十余个独立图层"）。
- 一篇文章通常 2-4 个这样的框,数量对应文章的核心论点数,不是装饰,是内容大纲的外化。

### 6. 图片块与视频块(都必须标来源)

原始标记(图片):
```html
<div class="photo-block reveal">
  <img loading="lazy" src="..." alt="南极秦岭站科考信息图，含时间轴、柱状图、饼图、折线图与站体实景">
  <div class="photo-cap">「南极秦岭站」信息图：一张图里排下科考时间轴……来源：字节跳动Seed</div>
</div>
```

原始标记(视频,复用同一套 caption 样式):
```html
<div class="video-block reveal">
  <video controls preload="metadata" playsinline src="..."></video>
  <div class="photo-cap">演示：整张海报拆成十余个透明图层……来源：字节跳动Seed</div>
</div>
```

写作规则:
- **图片/视频说明文字必须以"来源：XXX"结尾**——这一条零容忍,不标来源不发。
- `alt` 属性写实际描述(无障碍 + SEO),不是留空。
- 这一节的机制在 zhishare 已经用工具详情页的 `ToolMediaGallery`(2026-07-09 建的)实现过一次,文章详情页复用同一套组件与存储模式(`article-media/{slug}.json`,不是新发明)。

### 7. 相关推荐(文末)

原始标记:每张卡一个 `--tint` 强调色、分类角标、标题、日期·阅读时长。

zhishare 现状:`article-detail-related-card` 已经有类似结构(分类 tag + 标题 + 摘要 + 链接),**不需要重做**,只需确认视觉语言与本标准的其余部分统一(暖色调、衬线标题)。

## 可选组件(不强制,按内容需要使用)

以下三个来自 xiaohu 但**不适合 zhishare 内容形态**,降级为可选,不是标准的必修部分:

| 组件 | xiaohu 原用途 | zhishare 何时用 |
| --- | --- | --- |
| 代码/参数示例块(原 `.snippet`,`<pre>` + 标签) | 展示 AI 生图的完整 Prompt | 只有涉及命令行、配置代码、脚本片段的文章才用(如"自动化脚本"类工具教程) |
| 多图对比网格(原 `.imgrid`/`.fourgrid`) | 一次性铺开多张生成结果对比 | 只有需要并排比较多张截图时才用(如"5 个工具对比"类文章) |
| 横向滚动案例墙(原 `.showcase`,文末链官方案例页) | 引导去官方产品页看更多案例 | 只有被报道对象自己有丰富官方案例库时才用,大多数 zhishare 文章不需要 |

## 数据模型影响(仅记录,不在本文件实施)

当前 `ArticleSection` 类型(`src/components/articles/article-content.ts`)只有 `paragraphs` 和 `list` 两种,不足以承载上述结构。落地本标准需要扩展该类型并改造 `parseArticleSections`/`ArticleContentSection` 渲染逻辑,新增:

- 小节级 `lead`/`big` 双段导语字段
- `why` 类型(全文一次)
- `keypoint` 类型(tag + 加粗正文)
- `photo`/`video` 类型(复用 `ToolMediaGallery` 的存储与白名单嵌入模式,重命名为通用的 `ArticleMediaGallery` 或直接扩展现有组件)
- 可选:`snippet`(代码块)、`imgrid`(多图网格)、`showcase`(横向滚动)

这是一次真正的功能开发(类型改造 + 内容解析 + 渲染组件 + 至少一篇真实文章按新结构重写做验证),按项目惯例应该走 `brainstorming → 设计确认 → 写 spec → 写 plan → 执行` 的完整流程,是下一步的独立任务,不在本次研究范围内直接动代码。

## v2 补充(2026-07-16,样本:thinking-machines-inkling-open-model)

第二个样本证实了 v1 的全部规则(速览框、编号小节、lead 导语、来源披露、图注带来源),并暴露出 6 个 v1 未覆盖的格式元素。按对 zhishare 的价值排序:

### v2-1. 小节内的「问句式」子标题(写作规则,立即生效)

样本在编号 H2 小节内再分 H3 子标题,且大量用**读者视角的问句/口语**:「那么,Inkling 到底是什么?」「它和普通开放模型有什么不一样?」「下载得到,不代表普通电脑跑得动」。这是把"读者心里的下一个疑问"直接写成标题,而不是功能罗列。

zhishare 适配:长小节(4 段以上)内允许插入子标题块,标题优先写成读者会问的问题。渲染层需要新增 `subheading` 块类型。

### v2-2. 数据面板(渲染组件缺口,最大的一项)

样本正文里高频出现手工构建的**行内信息图**,不是贴截图,是 HTML/CSS 组件:

- **大数字统计卡**(`ik-stat`):数字放大 + 下方小字标签,两三张并排(如「9750 亿 总参数」vs「410 亿 每 token 激活」)。
- **好/坏对比卡**(`good`/`bad` 类):两栏并排,一栏"这样做对"一栏"这样做错"。
- **键值参数表**(`tc-row`):左键右值的紧凑参数/规格对照。
- **步骤轨道**(`ik-step`/`ik-track`):横向 3-5 步流程,每步一个短语。

zhishare 适配:这四种对"工具对比/避坑"内容形态几乎量身定做(参数表→工具规格,好坏卡→避坑,步骤→上手流程,大数字→免费额度/限制)。落地需要新增 4 个块类型:`stats`、`contrast`、`kv`、`steps`。

### v2-3. 「会教你读图」的图注(写作规则,立即生效)

v1 只要求图注以「来源:XXX」结尾。样本的图注更进一步:**先教读者怎么读这张图、有什么坑,再标来源**。例:「不同测试的分数被统一换算到 0–100;缺少公开成绩的项目按 0 绘制,因此它适合观察能力覆盖面,不适合用面积直接宣布谁是最强模型。来源:Thinking Machines Lab」。

zhishare 规则升级:数据类配图的图注必须包含「怎么读 + 边界/坑 + 来源」三件事;纯示意图可只写「内容 + 来源」。

### v2-4. 结尾收束句(写作规则,立即生效)

全文最后一个元素是一句放大的**总结判断句**(样本:「Inkling:从只交一份权重,变成模型、训练和部署连成一条路」),给读者一句可以带走转述的话。zhishare 适配:每篇文章以一个 `takeaway` 结尾块收束,一句话,有立场,不超过 40 字。渲染层需要新增 `takeaway` 块类型。

### v2-5. 标题带具体数字(写作规则,立即生效)

样本主标题:「每处理一个 token,只启用 410 亿参数」——用具体数字制造信息密度。zhishare 规则:标题里能放具体数字的就放(「4 件事」「5 个工具」已在这么做,保持)。

### v2-6. 视频占位(暂不采纳)

样本有自绘的视频交互示意(`ik-video`),zhishare 已有 bilibili/youtube 白名单 iframe 方案,不新增。

### v2 落地状态

- 写作规则(v2-1 问句标题、v2-3 教读图图注、v2-4 收束句、v2-5 数字标题):**即日起写/改文章时执行**。
- 渲染组件(`subheading`/`stats`/`contrast`/`kv`/`steps`/`takeaway` 六个新块类型):待实施,需类型扩展 + 解析器 + 渲染 + CSS + 一篇真实文章验证,按项目惯例走完整流程,见任务清单。
